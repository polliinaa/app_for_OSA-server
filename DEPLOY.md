# DEPLOY.md

## 🏗 Сетевое решение (Связь контейнера и базы)

Для выполнения задания настроено взаимодействие между Go-приложением в Docker-контейнере и СУБД PostgreSQL, установленной непосредственно на хостовой операционной системе (виртуальной машине).

### Проблема и реализация

По умолчанию Docker-контейнер изолирован от сетевых интерфейсов хоста. Для решения этой задачи применены следующие настройки:

1. **Настройка PostgreSQL на хосте:**
   - В файле `/etc/postgresql/16/main/postgresql.conf` изменен параметр `listen_addresses = '*'`.
   - В файле `/etc/postgresql/16/main/pg_hba.conf` добавлена строка для доступа из подсети Docker: `host all all 172.17.0.0/16 md5`.
   - Команда перезапуска для применения настроек: `sudo systemctl restart postgresql`.

2. **Организация доступа из контейнера:**
   - При запуске контейнера в GitHub Actions использован флаг `--add-host=host.docker.internal:host-gateway`. Это создает внутри контейнера адрес-указатель (`host.docker.internal`) на основную операционную систему.
   - В приложении через переменную окружения `DB_HOST` указано это имя, что позволяет Go-сервису подключаться к базе данных хоста по IP-адресу `192.168.1.36`.

## 📦 Описание процесса деплоя (Deployment)

### Почему выбран Self-hosted Runner?

Первоначальный вариант деплоя через стандартные облачные раннеры GitHub и SSH-секреты (Secrets) оказался невозможен из-за сетевых ограничений: виртуальная машина находится за файрволом и недоступна для входящих соединений извне.

**Выбранное решение:** установка локального агента (**Self-hosted runner**) прямо на целевую ВМ. Агент сам обращается к GitHub за обновлениями, что исключает необходимость открытия 22 порта (SSH) для интернета.

### Сборка и автоматизация

1. **Dockerfile (Multistage):** Использована двухэтапная сборка. На первом этапе (`builder`) в образе `golang:1.24-alpine` происходит компиляция бинарного файла. На втором этапе готовый файл копируется в чистый образ `alpine:latest`. Это минимизирует вес образа и повышает безопасность.
2. **CI/CD Pipeline:** При выполнении `git push` в ветку `main` локальный агент выполняет следующие команды:
   - `actions/checkout` — получение кода.
   - `docker build -t my-app .` — сборка образа.
   - `docker stop my-app || true` — остановка старого контейнера.
   - `docker run -d ...` — запуск обновленной версии с пробросом порта `8080` и сетевыми флагами.

## 🛠 Команды управления и мониторинга

Для стабильной работы и отладки сервиса на виртуальной машине используются два параллельных терминала:

### 1. Управление жизненным циклом деплоя (Терминал №1)

В данном окне запущен агент GitHub Actions, который обеспечивает автоматическую сборку при поступлении новых push-событий в репозиторий.

- **Запуск агента:** `~/actions-runner/run.sh`
- **Статус:** Состояние `Listening for jobs` подтверждает готовность сервера к приему команд на пересборку Docker-образа.

### 2. Мониторинг состояния системы и логирование (Терминал №2)

Используется для проверки сетевой связности и контроля работы запущенного контейнера.

- **Просмотр логов приложения:** `docker logs -f my-app`  
  _(Позволяет в реальном времени отслеживать статус подключения Go-сервиса к PostgreSQL на хосте)._
- **Проверка сетевых интерфейсов:** `ip a`  
  _(Подтверждает наличие адреса `192.168.1.36` и доступность интерфейса `docker0`)._
- **Проверка активных контейнеров:** `docker ps`  
  _(Отображает статус работы контейнера `my-app` и проброс порта `8080`)._
- **Тестирование эндпоинта:** `curl http://localhost:8080/hello`  
  _(Локальная проверка доступности API после завершения пайплайна)._
